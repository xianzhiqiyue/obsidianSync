# Obsidian 自建同步系统测试与验收计划（v1）

## 1. 测试目标
- 验证“最终一致、不丢数据、可恢复”三条底线。
- 证明需求文档中的 AC（验收标准）可被重复验证。

## 2. 测试范围
- 客户端：扫描、同步、冲突处理、重试、错误提示。
- 服务端：鉴权、变更提交、冲突判定、幂等、恢复接口。
- 系统级：多端并发、弱网、断网、服务重启、备份恢复。

## 3. 测试分层

| 层级 | 占比建议 | 内容 |
| --- | --- | --- |
| 单元测试 | 50% | 变更计算、CAS 判定、冲突命名、重试策略 |
| 集成测试 | 30% | API + DB + 对象存储协同 |
| 端到端测试 | 20% | 实机/模拟器多端同步与异常恢复 |

### 3.1 当前自动化入口
- 插件单元测试：`npm run --workspace @obsidian-sync/plugin test`
  - 覆盖 `sync-planner` 的失败队列重放、裁剪回退、rename/move 检测、旧数据兼容。
  - 覆盖 `sync-retry` 的弱网重试、部分上传失败恢复、超时中断重试。
  - 覆盖 `sync-notify` 的通知节流与失败阈值策略。
  - 覆盖 `state-store` 的阻塞态与手动清理恢复。

## 4.1 自动化脚本映射
- `scripts/smoke-api.sh`
  - 覆盖基础链路：`health/login/create-vault/state/prepare/commit/pull/download-urls`。
- `scripts/regression-sync-api.sh`
  - 覆盖冲突与一致性回归：
    - 并发更新触发 `VERSION_CONFLICT`。
    - rename/move 事件保留 `file_id`。
    - commit 幂等 key 重放返回相同响应。
- `scripts/drill-backup-restore.sh`
  - 覆盖运维演练回归：
    - 生成真实同步数据后执行 PostgreSQL + MinIO 备份。
    - 恢复到隔离库/桶并校验 checkpoint 连续性与对象可读性。
- `infra/docker-compose.monitoring.yml`
  - 覆盖监控模板验收：
    - Prometheus 能抓到 `sync-api` 指标。
    - Grafana 自动加载 `Obsidian Sync API Overview` dashboard。
- `scripts/load-sync-api.mjs`
  - 覆盖性能压测基础能力：
    - 多 worker 并发 `login/create-vault/prepare/upload/commit/pull`。
    - 记录 p50/p95/p99、错误率、重试次数、吞吐。
    - 支持弱网注入（延迟/超时/模拟丢包）。
- `scripts/run-perf-suite.sh`
  - 一键执行并发/弱网两个场景并输出报告：
    - `*-concurrency-baseline.json`
    - `*-weak-network.json`
    - `*-summary.md`
- `scripts/pre-beta-check.sh`
  - 覆盖 Beta 发布前最小检查：
    - build + plugin test + smoke + regression + metrics。
    - 可选追加快速压测（`RUN_PERF=1`）。
- `scripts/package-plugin-beta.sh`
  - 覆盖 Beta 产物打包：
    - 输出插件目录、zip 安装包、sha256、release 元数据。
- `scripts/new-beta-feedback.sh`
  - 覆盖反馈收集落盘：
    - 按 release/wave 生成标准化反馈记录。
- `scripts/summarize-beta-feedback.mjs`
  - 覆盖反馈汇总：
    - 输出严重级别、状态统计与未关闭 P0/P1 列表。
- `scripts/check-wave-gate.sh`
  - 覆盖放量门禁：
    - 指标检查 + 高优反馈检查综合判定是否可进入下一波。
- `scripts/init-beta-wave.sh`
  - 覆盖灰度执行初始化：
    - 生成 wave1/wave2/wave3 的参与者清单、每日记录与执行说明模板。
- `scripts/update-beta-feedback.mjs`
  - 覆盖反馈生命周期更新：
    - 按反馈 ID 或文件路径更新状态、严重级别、处理人、关联任务。
    - 自动追加处理记录并更新 front matter 的 `updatedAt`。
- `scripts/advance-beta-wave.sh`
  - 覆盖波次推进决策留痕：
    - 调用 `check-wave-gate` 进行门禁判定并落盘到 `wave-decisions.md`。
    - 通过时生成下一波 `ready-for-rollout.md`。

## 5. 异常与鲁棒性测试
- 弱网：高延迟（300-800ms）、高丢包（5-20%）。
- 服务异常：API 重启、对象存储短暂不可用、数据库只读。
- 时钟偏移：设备时间前后偏移 5-10 分钟。
- 大批量变更：一次 1000+ 文件修改。
- 客户端执行治理：同步超时中断、运行中重复触发合并排队。

## 6. 性能测试
- 指标目标：
  - 100 文件增量同步，P95 < 5s。
  - 10 设备并发提交，错误率 < 1%。
- 压测场景：
  - 写多读少（批量编辑后同步）。
  - 读多写少（多设备只拉取变更）。

## 7. 安全测试
- token 过期与刷新链路。
- 设备撤销即时生效验证。
- 越权访问测试（跨 Vault 访问）。
- TLS 配置检查（禁用明文、证书有效性）。

## 8. 验收流程
1. 执行 AC 对应用例并生成测试报告。
2. 覆盖 macOS + Windows + Android 三端实测。
3. 完成一次备份恢复演练并记录 RTO/RPO。
4. 由负责人签署“可 Beta 发布”结论。

## 9. 退出标准
- 所有 AC 用例通过。
- 无 P0/P1 未解决缺陷。
- 性能指标达标或有书面豁免。
- 恢复演练通过且文档齐全。
