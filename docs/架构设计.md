# Obsidian 自建同步系统架构设计（v1）

## 1. 设计目标
- 让 `macOS / Windows / Android` 端在弱网和并发场景下稳定同步。
- 采用“可演进”的分层架构：先保证正确性与恢复能力，再优化实时性。
- 将“文件内容”和“同步元数据”解耦，降低后续扩容与迁移成本。

## 2. 总体架构

```text
+--------------------+      HTTPS       +------------------------+
| Obsidian Plugin    | <--------------> | API Gateway / Sync API |
| (Desktop/Android)  |                  +-----------+------------+
+---------+----------+                              |
          |                                         |
          | local queue/state                       |
          v                                         v
+--------------------+                 +-------------------------+
| Local Vault Files  |                 | Auth + Sync Services    |
+--------------------+                 +-----+-----------+-------+
                                            |           |
                                            |           |
                                            v           v
                                   +----------------+ +------------------+
                                   | PostgreSQL     | | Object Storage   |
                                   | metadata/state | | (S3-compatible)  |
                                   +----------------+ +------------------+
```

## 3. 核心组件职责

| 组件 | 职责 |
| --- | --- |
| Plugin Sync Engine | 扫描本地变更、拉取远端变更、冲突处理、重试控制 |
| Plugin Local State | 保存 checkpoint、文件版本映射、失败队列 |
| API Gateway | 统一鉴权、限流、请求日志、路由 |
| Auth Service | 设备注册、token 刷新、设备撤销 |
| Sync Service | 变更校验、版本推进、冲突检测、提交事务 |
| Metadata DB | 存储 Vault/文件/版本/变更日志/tombstone |
| Object Storage | 存储文件内容对象（按 content hash 去重） |
| Observability | 指标、日志、告警、审计记录 |

## 4. 关键流程

### 4.1 首次同步
1. 客户端获取本地扫描结果并判断 vault 是否为空。
2. 用户显式选择 `Upload-First` 或 `Download-First`。
3. 服务端记录初始 checkpoint，禁止未决策略下自动双向同步。

### 4.2 增量同步（单次循环）
1. 客户端上传本地变更摘要（file_id/path/op/base_version/content_hash）。
2. 服务端校验 base_version（CAS）。
3. 对通过校验的变更发放上传地址（仅对缺失 object）。
4. 客户端上传对象并提交 commit。
5. 服务端事务化写入变更、推进版本、返回新 checkpoint。
6. 客户端按 checkpoint 拉取远端变更并应用本地。

### 4.3 冲突处理
1. CAS 失败返回 `VERSION_CONFLICT`。
2. 客户端拉取最新远端文件与本地版本比较。
3. 生成冲突副本（保留本地编辑内容），远端版本落地为主文件。
4. 记录冲突事件，用于后续优化策略。

## 5. 数据架构

### 5.1 关键实体
- `vaults`：Vault 基本信息与拥有者。
- `devices`：设备注册信息与状态（active/revoked）。
- `file_entries`：逻辑文件（file_id、current_path、head_version、deleted_at）。
- `file_versions`：历史版本（version、content_hash、author_device、created_at）。
- `changesets`：一次提交的变更集合（checkpoint、device_id、status）。
- `tombstones`：删除墓碑（file_id、deleted_at、expire_at）。

### 5.2 一致性策略
- 元数据变更使用数据库事务提交。
- 对象存储写入先于 metadata commit。
- commit 失败时对象可保留，后续由 GC 回收孤儿对象。

## 6. 部署拓扑

### 6.1 单机部署（MVP）
- `nginx`（TLS 终止）+ `sync-api` + `postgres` + `minio`。
- 使用 Docker Compose 管理，便于快速上线和备份。

### 6.2 可扩展部署（生产）
- API 无状态化，多副本水平扩容。
- PostgreSQL 主从或托管服务。
- 对象存储切换到托管 S3 或分布式 MinIO。
- 可选 Redis 缓存热点元数据与短期 token 黑名单。

## 7. 安全架构
- 强制 HTTPS，拒绝明文 HTTP 请求。
- access token 短有效期，refresh token 可撤销。
- 每个 Vault 使用独立逻辑密钥域，避免横向访问。
- 审计日志记录关键操作：登录、设备注册、设备撤销、批量删除。

## 8. 可观测性设计
- 指标：
  - `sync_success_rate`
  - `sync_request_p95_ms`
  - `conflict_rate`
  - `retry_exhausted_count`
- 日志：
  - 请求日志（trace_id）
  - 同步日志（vault_id、device_id、checkpoint）
  - 错误日志（stack + request context）
- 告警：
  - 5 分钟错误率 > 5%
  - P95 延迟持续高于阈值
  - 对象存储写入失败率异常

## 9. 可靠性与恢复
- 每日自动备份 PostgreSQL（全量 + WAL/增量）。
- 对象存储开启版本化或定期快照。
- 每月至少 1 次恢复演练，验证 RPO/RTO。
- tombstone 保留期默认 30 天，支撑误删恢复。

## 10. 技术选型建议
- 服务端：`Go`（并发与部署简单）或 `Node.js + TypeScript`（生态丰富）。
- 数据库：`PostgreSQL 16+`。
- 对象存储：`MinIO`（自建）或云厂商 S3。
- 网关与 TLS：`Nginx` 或 `Caddy`。

## 11. 风险与缓解
- 风险：移动端后台限制导致同步中断。  
  缓解：前台触发优先 + 失败重试 + 明确用户提示。
- 风险：并发冲突高。  
  缓解：CAS + 冲突副本 + 冲突指标监控。
- 风险：误删扩散。  
  缓解：tombstone + 回收延迟 + 快速恢复流程。
