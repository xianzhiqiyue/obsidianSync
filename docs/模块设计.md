# Obsidian 自建同步系统模块设计（v1）

## 1. 设计原则
- 单一职责：每个模块只负责一类问题，接口清晰。
- 幂等优先：同步请求允许重试，不因重复请求破坏状态。
- 可替换：插件端与服务端模块通过接口契约解耦。

## 2. 插件端模块

| 模块 | 主要职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| `VaultScanner` | 扫描本地文件变化，生成本地变更集 | vault 文件树、上次快照 | `LocalChange[]` |
| `LocalStateStore` | 保存 checkpoint、文件映射、任务队列 | 同步结果、任务状态 | 本地状态读写接口 |
| `SyncPlanner` | 规划上传与下载步骤，合并批次 | 本地变更、远端状态 | `SyncPlan` |
| `SyncTransport` | 执行 API 调用、上传下载对象 | `SyncPlan` | API 响应、错误 |
| `ConflictResolver` | 处理版本冲突并生成冲突副本 | 冲突响应、本地内容 | 冲突文件、事件日志 |
| `ApplyEngine` | 将远端变更应用到本地 vault | `RemoteChange[]` | 本地文件操作结果 |
| `RetryController` | 指数退避重试和失败终止策略 | 失败任务 | 新的重试任务或终止 |
| `SettingsUI` | 服务地址、登录、策略、诊断信息 | 用户输入 | 配置更新 |
| `Diagnostics` | 记录调试日志与同步摘要 | 同步过程事件 | 本地日志、错误提示 |

### 2.1 插件端状态机

```text
IDLE
  -> SCAN_LOCAL
  -> PLAN_SYNC
  -> PUSH_LOCAL
  -> PULL_REMOTE
  -> APPLY_REMOTE
  -> COMMIT_LOCAL_STATE
  -> IDLE

任何状态失败 -> RETRY_WAIT -> PLAN_SYNC
重试耗尽 -> ERROR_BLOCKED（等待用户手动恢复）
```

### 2.2 插件端关键接口（TypeScript 草案）

```ts
interface LocalChange {
  fileId?: string;
  path: string;
  op: "create" | "update" | "delete" | "rename" | "move";
  baseVersion?: number;
  contentHash?: string;
  ts: number;
}

interface SyncPlan {
  uploadChanges: LocalChange[];
  pullFromCheckpoint: string;
}

interface SyncEngine {
  runOnce(): Promise<void>;
  runWithRetry(): Promise<void>;
}
```

## 3. 服务端模块

| 模块 | 主要职责 | 依赖 |
| --- | --- | --- |
| `AuthModule` | 登录、token 刷新、设备注册/撤销 | PostgreSQL、JWT |
| `VaultModule` | Vault 访问控制、成员关系校验 | PostgreSQL |
| `SyncModule` | 变更校验、checkpoint 推进、冲突判定 | PostgreSQL、ObjectStore |
| `ObjectModule` | 对象上传地址签发、完整性校验、GC 标记 | ObjectStore |
| `ConflictModule` | 记录冲突事件和统计 | PostgreSQL |
| `AuditModule` | 审计日志写入与查询 | PostgreSQL |
| `RateLimitModule` | 限流和防滥用 | Redis（可选） |
| `MetricsModule` | 指标导出和 tracing | Prometheus/OpenTelemetry |

### 3.1 服务端调用链
1. Gateway 校验 token 和限流。
2. `SyncModule` 校验 vault 权限与 base_version。
3. `ObjectModule` 返回上传地址，客户端上传对象。
4. `SyncModule` 事务提交 changeset，推进 checkpoint。
5. `AuditModule` 与 `MetricsModule` 记录事件。

## 4. 数据访问层设计
- Repository 模式：
  - `VaultRepo`
  - `DeviceRepo`
  - `FileEntryRepo`
  - `FileVersionRepo`
  - `ChangesetRepo`
  - `TombstoneRepo`
- 所有跨表写操作必须通过 Service 层事务封装。

## 5. 关键业务规则落地
- `delete` 必须先写 tombstone，再更新 head_version。
- `rename/move` 必须是显式 op，保留 file_id 不变。
- `commit` 必须保证：
  - base_version 校验通过。
  - object_hash 存在且校验通过。
  - changeset 与 checkpoint 原子推进。

## 6. 失败场景处理

| 场景 | 处理策略 |
| --- | --- |
| 上传对象成功但 commit 失败 | 客户端可重试 commit；服务端按 idempotency key 去重 |
| commit 成功但客户端超时 | 客户端重放同一 idempotency key，服务端返回已提交结果 |
| 拉取变更中断 | 下次从上次 checkpoint 继续 |
| 本地应用远端变更失败 | 记录错误并暂停自动同步，提示用户恢复 |

## 7. 模块边界约束
- 插件端不直接操作数据库，只通过 API。
- 服务端不感知 Obsidian UI，仅提供同步能力。
- 冲突判定在服务端，冲突副本生成在客户端。

## 8. 可扩展点
- 后续可增加 `RealtimeModule`（WebSocket 推送变更通知）。
- 后续可增加 `ChunkModule`（大文件分块上传与断点续传）。
- 后续可增加 `E2EEncryptModule`（端到端密钥管理）。
