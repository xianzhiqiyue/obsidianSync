# Obsidian 自建同步系统部署与运维手册（v1）

## 1. 部署目标
- 可快速在 `VPS` 或家庭服务器上线。
- 支持 HTTPS、安全鉴权、备份恢复与告警。

## 2. 组件清单
- `gateway`：Nginx/Caddy（TLS 终止、反向代理）
- `sync-api`：同步服务
- `postgres`：元数据库
- `minio`：对象存储（或直接使用云 S3）
- `prometheus + grafana`（可选）：监控与可视化

## 3. 环境要求
- CPU：2 vCPU 起步
- 内存：4GB 起步
- 存储：50GB SSD 起步（按 Vault 体量扩容）
- OS：Linux x86_64 / ARM64 均可
- 网络：可访问 443 端口

## 4. 配置项（最小集）
- `APP_ENV=prod`
- `BASE_URL=https://sync.example.com`
- `JWT_SECRET=...`
- `ACCESS_TOKEN_TTL=1h`
- `REFRESH_TOKEN_TTL=30d`
- `POSTGRES_DSN=...`
- `S3_ENDPOINT=...`
- `S3_BUCKET=obsidian-sync`
- `S3_ACCESS_KEY=...`
- `S3_SECRET_KEY=...`
- `LOG_LEVEL=info`

## 5. 部署步骤（Docker Compose）
1. 准备 `.env` 配置文件。
2. 启动依赖：`postgres`、`minio`。
3. 执行数据库迁移。
4. 启动 `sync-api`。
5. 配置网关 TLS 与反代到 `sync-api`。
6. 执行健康检查与首次连通测试。

### 5.1 生产端口与凭据加固（推荐）
- 复制生产环境变量模板：
  - `cp infra/.env.prod.example infra/.env.prod`
- 至少修改以下字段为强随机值：
  - `POSTGRES_PASSWORD`
  - `MINIO_ROOT_USER`
  - `MINIO_ROOT_PASSWORD`
- 显式设置端口绑定策略（建议）：
  - `POSTGRES_PORT_BIND=127.0.0.1:5432:5432`
  - `MINIO_API_PORT_BIND=0.0.0.0:9000:9000`
  - `MINIO_CONSOLE_PORT_BIND=127.0.0.1:9001:9001`
- 使用生产叠加编排启动：
  - `docker compose --env-file infra/.env.prod -f infra/docker-compose.yml -f infra/docker-compose.prod.yml up -d`
- 说明：`infra/docker-compose.prod.yml` 仅用于生产通用策略（例如 `restart`），端口绑定统一由 `.env.prod` 驱动，避免 compose 叠加时端口列表合并歧义。
- 端口暴露策略：
  - `5432` 仅本机监听（数据库不暴露公网）。
  - `9000` 对外开放（预签名上传/下载需要）。
  - `9001` 仅本机监听（MinIO Console 不暴露公网）。
- 建议同时启用主机防火墙（`ufw`）二次收口：
  - 预览规则：`scripts/harden-firewall.sh`
  - 应用规则：`APPLY=1 scripts/harden-firewall.sh`
  - 脚本默认同步写入 `DOCKER-USER` 丢弃规则，防止 Docker 端口转发绕过 `ufw` 的 `INPUT` 链策略。

## 6. 证书与 HTTPS
- 强制 HTTPS，HTTP 仅做 301 跳转。
- 证书建议使用 Let's Encrypt 自动续期。
- 即使在 Tailscale 网络内，也保持 TLS，不降级到明文。

## 7. 备份策略

### 7.1 数据库
- 每日全量备份。
- 每小时增量/WAL 归档（可选）。
- 保留策略：最近 7 天全量 + 30 天增量。

### 7.2 对象存储
- 开启版本化（优先）或每日快照。
- 关键 bucket 做跨盘/跨节点复制（可选）。

### 7.3 恢复演练
- 每月至少 1 次在独立环境恢复并验证：
  - 能拉起服务。
  - 能读取历史文件。
  - 能继续同步。

### 7.4 脚本化备份（仓库内置）
- PostgreSQL 备份：
  - `scripts/backup-postgres.sh`
  - 关键环境变量：
    - `POSTGRES_MODE=auto|local|docker`（默认 `auto`）
    - `BACKUP_DIR`、`BACKUP_TAG`
- PostgreSQL 恢复（破坏性）：
  - `ALLOW_DROP=1 POSTGRES_BACKUP_FILE=<path> TARGET_DB_NAME=<db> scripts/restore-postgres.sh`
- MinIO 备份：
  - `scripts/backup-minio.sh`
  - 关键环境变量：
    - `MC_MODE=auto|local|docker`（默认 `auto`）
    - `S3_BUCKET`、`BACKUP_DIR`、`BACKUP_TAG`
- MinIO 恢复：
  - `MINIO_BACKUP_DIR=<dir> TARGET_S3_BUCKET=<bucket> REMOVE_EXTRA=1 scripts/restore-minio.sh`

### 7.5 一键恢复演练
- 执行：
  - `scripts/drill-backup-restore.sh`
- 演练脚本会自动完成：
  1. 登录并创建演练 Vault，提交一条真实同步变更。
  2. 备份 PostgreSQL 与 MinIO。
  3. 恢复到隔离目标库/目标桶。
  4. 校验恢复后 checkpoint 连续性与对象可读性。
- 清理策略：
  - 默认保留恢复目标用于人工核查。
  - `CLEANUP_VERIFY_TARGETS=1` 可在演练成功后自动清理。

## 8. 监控与告警

### 8.1 关键指标
- API 成功率、P95 延迟、5xx 比例
- commit 冲突率
- 重试耗尽次数
- DB 连接池、慢查询数量
- 对象存储可用性与错误率

当前服务已提供 Prometheus 指标端点：
- `GET /api/v1/metrics`

关键指标名（示例）：
- `sync_api_http_requests_total{method,route,status}`
- `sync_api_http_request_duration_seconds_bucket{method,route,le}`
- `sync_api_sync_prepare_total{result}`
- `sync_api_sync_prepare_conflicts_total`
- `sync_api_sync_commit_total{result}`
- `sync_api_sync_commit_applied_changes_total`
- `sync_api_sync_pull_total{result}`
- `sync_api_sync_pull_changes_total`
- `sync_api_auth_login_total{result}`
- `sync_api_auth_refresh_total{result}`

### 8.2 告警建议
- 5 分钟错误率 > 5%
- P95 > 2s 且持续 10 分钟
- DB 磁盘使用率 > 80%
- 备份任务失败

本地阈值检查脚本（可接入 cron/CI）：
- `scripts/check-metrics.sh`
- 可配置阈值：
  - `MAX_COMMIT_FAILED_TOTAL`
  - `MAX_HTTP_5XX_TOTAL`
  - `MAX_PREPARE_CONFLICTS_TOTAL`

### 8.3 可视化模板（仓库内置）
- Compose 叠加启动：
  - `docker compose -f infra/docker-compose.yml -f infra/docker-compose.monitoring.yml up -d`
- 配置文件：
  - Prometheus 抓取配置：`infra/monitoring/prometheus/prometheus.yml`
  - Grafana 数据源配置：`infra/monitoring/grafana/provisioning/datasources/prometheus.yml`
  - Grafana dashboard provider：`infra/monitoring/grafana/provisioning/dashboards/dashboard.yml`
  - dashboard 草案：`infra/monitoring/grafana/dashboards/obsidian-sync-api-overview.json`
- 默认访问地址：
  - Prometheus：`http://localhost:9090`
  - Grafana：`http://localhost:3001`

## 9. 日志规范
- 统一 JSON 日志。
- 每条日志含 `trace_id`、`vault_id`、`device_id`（可用时）。
- 敏感字段（token、密钥）必须脱敏。

## 10. 故障处理 Runbook

### 10.1 同步失败激增
1. 看 API 错误码分布（401/409/500）。
2. 检查对象存储和数据库健康状态。
3. 回看最近发布记录。
4. 必要时回滚上一版本并保留审计日志。

### 10.2 大量冲突
1. 检查客户端版本是否分裂。
2. 检查是否存在错误的 base_version 提交。
3. 临时降低自动同步频率并引导用户手动合并。

### 10.3 数据恢复
1. 停止写入流量。
2. 恢复数据库到指定时间点。
3. 挂载对应对象存储快照。
4. 验证 checkpoint 连续性后恢复服务。

## 11. 发布与回滚
- 发布前：
  - 执行迁移 dry-run。
  - 通过 smoke test（登录、prepare、commit、pull）。
- 发布后：
  - 观察 30 分钟核心指标。
- 回滚：
  - 应用层先回滚镜像。
  - 若涉及 schema 变更，需预置向后兼容迁移。

## 12. 运维检查清单
- 每日：服务可用性、错误率、备份结果。
- 每周：容量趋势、慢查询、异常设备行为。
- 每月：恢复演练、安全凭据轮换、依赖升级审查。

## 13. 压测基线（P4）
- 单场景压测：
  - `SCENARIO=concurrency-baseline USERS=10 DURATION_SEC=60 node scripts/load-sync-api.mjs`
  - `SCENARIO=weak-network WEAK_MODE=1 USERS=8 DURATION_SEC=60 node scripts/load-sync-api.mjs`
- 一键双场景：
  - `scripts/run-perf-suite.sh`
- 报告输出：
  - `reports/perf/*-concurrency-baseline.json`
  - `reports/perf/*-weak-network.json`
  - `reports/perf/*-summary.md`
