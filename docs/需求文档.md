# Obsidian 自建同步系统需求文档（v2）

## 1. 项目目标
构建一个可自托管的多端同步系统，支持 `macOS / Windows / Android` 的 Obsidian Vault 同步，优先保证稳定性与可恢复性，再逐步优化实时性与体验。

核心目标：
- 支持自建部署（VPS 或家庭服务器）并长期稳定运行。
- 兼容弱网、断网、跨设备并发编辑。
- 具备明确的安全模型（鉴权、加密、设备撤销）。
- 有可执行的插件分发、升级、排障流程。

## 2. 范围与非范围
本期范围（MVP + 可生产）：
- 自研 Obsidian 插件（桌面与移动端通用）。
- 自建后端同步服务（元数据 + 对象存储）。
- 可靠的增量同步与冲突处理。
- 基础运维、监控、备份与恢复方案。

本期非范围：
- 完全复刻官方 Obsidian Sync 的实时协同体验。
- 文件级以下的实时协作 CRDT。
- iOS 首发支持（后续扩展）。

## 3. 技术路线决策

### 3.1 备选方案
方案 A：从零复刻官方 Sync 全量能力  
方案 B：标准化同步能力 + 自建后端（推荐）

### 3.2 采用方案
采用方案 B，具体落地为：
- 客户端：自研插件（统一桌面和 Android 行为）。
- 协议：基于 `checkpoint + 变更集` 的 REST 同步协议（简化且可演进）。
- 存储：对象存储（S3 兼容）保存文件内容，关系库保存元数据。
- 分发：先采用手动安装，后续接入发布渠道自动检查更新。

选择理由：
- 实现复杂度显著低于“全量仿官方”。
- 可测试性和可观测性更好，便于快速迭代。
- 保留后续演进空间（分块传输、实时推送、端到端增强）。

## 4. 关键术语
- `Vault`：一个独立知识库。
- `Device`：一个安装插件并接入同步的设备实例。
- `Object`：文件内容对象，以内容哈希标识（去重）。
- `Change`：一次文件操作（新增/修改/删除/重命名/移动）。
- `Checkpoint`：服务端确认的同步位置，用于增量拉取。
- `Tombstone`：删除墓碑，避免已删文件被旧设备“复活”。

## 5. 功能需求

### 5.1 账号与设备
- FR-01：支持用户登录与 Vault 维度隔离。
- FR-02：支持设备注册、设备命名、设备撤销。
- FR-03：设备撤销后 token 立即失效。

### 5.2 同步能力
- FR-04：首次同步支持两种模式：
  - `Upload-First`：本地作为主源上传。
  - `Download-First`：空库从云端全量下载。
- FR-05：增量同步基于 checkpoint 拉取远端变更。
- FR-06：上传支持幂等提交，失败可安全重试。
- FR-07：支持新增、修改、删除、重命名、移动。
- FR-08：删除必须写入 tombstone，并设置保留期。
- FR-09：弱网下自动重试（指数退避 + 最大重试次数）。

### 5.3 冲突与恢复
- FR-10：并发修改冲突时创建冲突副本，不覆盖原文件。
- FR-11：冲突副本命名规范统一（含设备名与时间戳）。
- FR-12：支持版本历史回滚（至少保留近 N 版或 N 天）。
- FR-13：支持误删恢复（基于 tombstone + 历史对象）。

### 5.4 插件与配置
- FR-14：插件支持配置服务地址、登录、Vault 绑定、同步策略。
- FR-15：支持手动触发同步与前台定时同步。
- FR-16：展示最近同步状态、错误原因、重试入口。

### 5.5 分发与升级
- FR-17：桌面端支持通过 `.obsidian/plugins/<plugin-id>/` 安装。
- FR-18：Android 支持同路径手动安装与升级说明。
- FR-19：插件支持“有新版本可用”提示（不强制自动升级）。

## 6. 非功能需求

### 6.1 安全
- NFR-01：所有网络请求必须使用 HTTPS（含 Tailscale 场景）。
- NFR-02：采用短期 access token + 长期 refresh token。
- NFR-03：服务端不存储明文用户主密码。
- NFR-04：敏感配置（token、密钥）本地安全存储。

### 6.2 性能
- NFR-05：常规增量同步（<100 个变更）目标 5 秒内完成。
- NFR-06：重复内容通过对象哈希去重，减少上传流量。

### 6.3 可靠性
- NFR-07：服务端可用性目标 `99.9%`（单区域）。
- NFR-08：同步接口具备幂等性与请求去重。
- NFR-09：异常退出后可基于本地队列恢复未完成任务。

### 6.4 可观测性
- NFR-10：记录请求日志、同步事件日志、关键错误栈。
- NFR-11：提供核心指标：成功率、P95 延迟、冲突率、失败重试率。

## 7. 同步规则（统一口径）

### 7.1 文件标识
- 逻辑文件以 `file_id` 标识，路径可变。
- 内容以 `content_hash` 标识，支持内容去重。

### 7.2 变更提交
- 客户端提交变更时携带 `base_version`（本地认知版本）。
- 服务端做 CAS 校验：
  - 一致则提交成功并生成新版本。
  - 不一致则返回冲突，客户端生成冲突副本并重拉远端。

### 7.3 删除与重命名
- 删除不立即硬删，写 tombstone。
- 重命名/移动必须作为显式事件，不可退化为“删+增”。

### 7.4 首次同步防冲突
- 非空库首次接入必须显式选择 `Upload-First` 或 `Download-First`。
- 未选择前禁止自动双向同步。

## 8. 平台与部署要求
- 支持部署模式：
  - 公网 VPS（Docker Compose）。
  - 家庭服务器 + Tailscale（仍使用 HTTPS）。
- Android 端要求：
  - 允许 Obsidian 前台稳定运行并尽量关闭电池优化。
  - 按实际机型校验存储权限，不将“管理所有文件”设为唯一前置条件。

## 9. 验收标准
- AC-01：三端可完成首次全量同步，文件一致率 100%。
- AC-02：三端并发编辑同一文件可稳定触发冲突副本，不丢数据。
- AC-03：断网恢复后可自动补齐增量变更。
- AC-04：删除/重命名/移动在多端最终一致。
- AC-05：设备撤销后旧 token 立即不可用。
- AC-06：可执行备份恢复演练并通过。
